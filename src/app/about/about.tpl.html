Welcome to Web Dev and Design

This course is about making websites. To that end, you'll be learning the technical stuff: HTML, CSS, and Javascript. At the same time, we'll also work on higher order, more conceptual stuff: elements of design, information architecture, usability, and purpose. Your work in the course will be centered around a series of projects, supported by readings and exercises to develop technical skills and design thinking you'll need for those projects. I think that the more this course can be focused on whatever it is you really want to make, the better it will be.

Learning to Code

Learning to code can be pretty different from the stuff you're used to learning in school. Here's some advice I've gathered from past students (and myself) over the years.

Set up your environment.
Coding almost always involves interacting with multiple files in multiple programs all at the same time. Invest some time in setting up your computer as an effective workspace. And, from time to time, rearrange it to suit your increasingly complex needs.
Experiment
To code effectively, you've got to understand how the code works in practice, what it actually does. Look at examples, change stuff and see what happens, even make little examples for yourself. Sure, coding is technical and conceptual, but it is also—maybe even more so—a craft. You can't learn carpentry just by studying the idea of a dovetail joint—you've got to try it. The same goes for coding.
Speed and Grammar: practice!
This is a subject where speed and grammar are both going to matter quite a bit. This is not some kind of silly, arbitrary measure of proficiency; it's a practical necessity. It will be hard to make genuine progress making things without being fairly fluent at writing code; and because computers, unlike (most) humans, are extremely unforgiving of grammatical mistakes. The only answer I know to this: type and type and type and type and type. Write lines upon lines of code—and write them yourself. (Actually, here are two other tips: learn keyboard shortcuts, and format your code.)

Learning Design

This course is as much about design thinking as it is about coding. Design principles certainly apply across different media, but, of course, we'll be focusing on design in the context of making websites. How?

Experiment
As with coding, experimentation is at the heart of design. Sometimes, you'll experiment specifically to understand how a certain bit of code works. Other times, you'll experiment with different aesthetics or user interaction design ideas. But ideally, you'll spend most of your time really playing with both at the same time.
Iterate
Ideas are cheap. It's what you do with them. Getting feedback and revising and then getting feedback and then revising and then getting feedback and then revising.... No need to belabor this point: critiques are an absolutely essential part of the design process, and participating in them from both sides is essential to developing your design thinking and skills.
Collect Inspriation
Just as important as making stuff is looking at (and playing with) stuff. Be a sponge, collect examples of design that you see—both on and offline—and then use it responsibly. Also, start a list of designers whose work you admire; look at their work, read about their processes on their blogs, try to run into them in coffee shops, ....

Freedom & Community

We'll be spending a lot of our class time in "lab time", in which you'll be free to experiment, work on exercises and projects, ask questions, and get informal feedback from me and each other. What's obvious is that this lab is a shared space, and people are here to learn. What's maybe less obvious is that students are coming to this class with a wide range of experience and comfort with computers, programming, and design. There's an art to being a badass without being intimidating, and to helping someone solve a problem in a way that also helps him/her learn. There's also an art to being a confident yet receptive novice.

Your Code & Other People’s Code

There is a lot of free code out there online. You can learn a lot from reading other people’s code. It can show you techniques for solving problems, examples of how to use new tools, models of clear and elegant code. (Of course, there’s also a lot of messy code out there, so beware.) So, yes: we will be reading plenty of code from other people in here—from the textbook, from each other, from the Internet.

But there’s a difference between reading other people’s code and using it. Certainly, if you use someone’s code, you should cite your source as a matter of intellectual and artistic honesty. Early on, though, I would strongly advise even against doing that: as mentioned above, you need practice actually typing lines of code yourself. So, for the first month, please don’t copy and paste code—not even if you have permission. Read other people's code to learn from it, but type your own. (The one exception to this is copying-and-pasting so that you can read the code in your own environment, and maybe tinker with it as you read.)

Finally: occasionally, there will be times when I ask you not to refer to any outside sources. I understand that this restriction is artificial, and that challenging assignments amidst heavy workloads can cause anxiety. If you are ever in such a bind that you think there’s no way out, talk to me. I will help.

What You Need

  • Reliable access outside of class to a computer with:
    • Internet access
    • Google Chrome
    • A code editor. We'll start with TextWrangler for now, but once you've got some of the basics down, Brackets and Sublime Text are more good options with a bit more power.
  • HTML & CSS: Design and Build Websites by Jon Duckett. Wiley. (2011). ISBN: 978-1-118-0081808
  • Eloquent Javascript: A Modern Introduction to Programming by Marijn Haverbeke. (2011). ISBN: 978-1593272821. (available free online)
  • A notebook or sketchbook.
  • Optional: a laptop to use in class, as long as it can run Chrome and your HTML editor of choice.
